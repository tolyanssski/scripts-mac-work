# Реализовать интеграцию с платежной системой (ПС) SmilePayz и всеми ее платежными методами

Эта платежная система предоставляет возможность приема платежей и осуществления выплат
через множество региональных платежных методов в поддерживаемых странах.
Предстоит добавить стандартный адаптер - новый пакет smilepayz внутри provider/adapter, 
99% всего нового кода будет там.
В соответствии со сложившейся практикой новый пакет адаптера должен внутри иметь одну базовую структуру
Adapter (сервис), который реализует общий интерфейс Adapter.

Тебе в рамках выполнения текущей задачи нужно будет следовать этому гайду при написании кода.
Не весь функционал, описанный в этом пакете, будет реально нужен в текущей задаче (часть функций не поддерживается SmilePayz),

Также внимательно изучи содержимое репозитория `~/projects/fonbet-payments-reference`.
Там лежит пример реального конфига из другой компании, где биллинг и интеграции пишутся почти так же как у нас.

## Важное уточнение по архитектуре

За пределами адаптера SmilePayz не должно упоминаться нигде в коде упоминание этой ПС (исключение: в конфиге, а также в некоторых случаях в работе с платежными инструментами).
Сами названия полей, необходимые для этой ПС, можно добавлять везде где нужно. 
Но за пределами адаптера эти названия полей должны выглядеть более абстрактными и общими, и по возможности
в будущем могут быть переиспользованы для других ПС.

**Пример:** 
Smilepayz может требовать поле PayerEmail.
Внутри адаптера мы его так и называем.
Но за пределами адаптера по возможности лучше придумать общее/универсальное название, например Email.
А если такое поле Email уже фигурирует там где нужно, и имеет тот же тип который нам нужен,
то переиспользуй его.

**Иными словами:**
adapter/smilepayz - пакет специфичный для этой пплатжной системы.
Пакет config может иметь необходимую конфигурацию этой ПС, а DI соответственно делает сборку app-контейнера.
Остальные пакеты - абстрактная логика, ничего не должны знать про эту ПС.

## Важное уточнение по хранению специфичных полей в сущности Transaction

Для этой ПС могут быть специфичные поля, требуемые для разных платежных методов и которые должны 
сохраняться в транзакцию.
Если такое требуется, тебе надо эти поля сериализовать в JSON и сохранять в поле AdditionalData 
сущности Transaction.
НО эти дополнительные данные НИГДЕ за пределами адаптера не называй по названию ПС. Так и называй additionalData.

## Соответствие терминологии SmilePayz и нашей внутренней, и особенности платежной системы

- У них: PAYIN / Pay-in ; у нас: Deposit
- У них: PAYOUT / Pay-out ; у нас: Payout либо Withdrawal
- У них: Payment method ; у нас: Channel. ВАЖНО: у нас тоже есть термин "платежный метод", но мы его будем считать отдельной сущностью со своим неймингом и конфигурацией, которая просто ссылается на конкретный Channel платежной системы
- У них: callback ; у нас: callback либо webhook

Особенности: в платежной системе SmilePayz нет поддержки работы с платежными инструментами.
Соответственно там где в коде будет требоваться создать/вернуть инструмент - надо возвращать заглушку с названием
платежного метода (примерно как в адаптере crypto).

## Начало работы (подготовка базового скелета кода и конфигурации)

### Скелет кода

Создай новый пакет smilepayz внутри adapter и сгенерируй заглушки для всех методов интерфейса.
Запили базовую структуру конфигурации этого адаптера, пробрось его в DI и пропиши в provider.

Для метода Deposit() внутри добавь проверку, что action должен быть только start. 
Action==confirm в нашей ПС не поддерживается.
При этом для Payout() поддерживается и start и confirm. Сделай проброс данных далее в приватные методы
для соответствующих action.

API base URL вынеси не только в конфиг, но и в env. Вместе с кредами. Потому что URL запросов к ПС будет разный в зависимости от 
test/prod среды.

### DTO со всеми возможными полями запросов к deposit / withdrawal

Нужно пройтись по примерам кода от SmilePayz по каждой стране и собрать в одну кучу 
все возможные поля для запроса к ПС. Отдельно для депозитов и отдельно для выплат сделай DTO (внутри адаптера smilepayz)
со всеми найденными полями в рамках всей SmilePayz.

ВАЖНО: эта DTO специфичная для ПС должна лежать внутри адаптера SmilePayz.
А для работы за пределами адаптера продублируй эти же поля в другую (уже общую) DTO PaymentAdditionalData.
Туда потом будем сваливать в общую кучу поля и других ПС.

То есть получится два DTO: один специфичный SmilePayz внутри адаптера SmilePayz,
второй общий в общей папке dto с абстрактным названием PaymentAdditionalData.

Внутри адаптера работа с локальным DTO, за пределами - с общим. При необходимости сделай функцию маппинга полей
(конвертации одного DTO в другое).

### Создай каналы для всех платежных методов ПС

Нужно вытащить списки платежных методов на строне ПС из json-файла `~/projects/smilepayz-demo-golang/payment_methods.json`.
Название метода на стороне ПС находится в ключе id в этом файле.
title - человекочитаемое название, а description - описание, поле country - страна в которой находится этот метод.
type - тип платежного метода в плане "депозит" или "выплата".
По каждому методу на строне ПС придумай название канала в виде <страна><название_метода_в_ПС>, 
а также название платежного метода на нашей стороне в формате <название_ПС_в_нижнем_регистре>_<2-буквенный_код_страны>_<название_метода_в_ПС_в_нижнем_регистре>.

Например: ты увидел там платежный метод PIX, относящийся к Бразилии.
В этом случае структура канала в Go будет называться `BrazilPIX`, у канала будет текстовый slug/ключ для
поиска - `brazil_pix`, а соответствующий платежный метод на нашей стороне будет называться `smilepayz_br_pix`.

Все каналы должны реализовывать один общий интерфейс Channel.
На данном этапе в каждом канале как минимум должен быть метод получения названия платежного метода на стороне ПС
(в случае с примером выше - именно строка PIX как он называется в SmilePayz).
Также метод для получения поддерживаемой страны (Brazil или BR в примере).
И еще запили методы для получения списков необходимых полей для депозита и выплаты.
Также сделай метод который возвращает то, какая валюта поддерживается данным методом на стороне ПС.
Возможно еще какая-то статичная конфигурация которая должна быть захардкожена в рамках каждого платежного канала - 
тоже должна возвращаться каналом в виде метода.

ВАЖНО: не складывай все каналы в один общий файл / структуру. Надо создать папку с каналами и складывать в нее
каналы в виде "один канал = один файл = одна самостоятельная структура".

### Создай YAML-конфигурацию для всех платежных методов, включая описание полей форм, которые требуются по каждому платежному методу

Пользуйся примерами из fonbet. Заполняй также человекочитаемые названия, описания, а также slugs, названия платежных
методов на нашей стороне (см выше где делали каналы), и маппинги названий платежных методов на нашей стороне
к соответствующим каналам.
В конфигурации также доплниельно для каждого платежного метода сделай новое поле 
supportedCountries (тип: массив строк).

ВАЖНО: полностью заполни yaml-конфиг smilepayz всеми методами и необходимыми для них полями.
Не бойся дублирования кода с каналами. Помни: в будущем на один канал может быть завязано 
несколько платежных методов на нашей стороне. И в коде у нас каналы, а в yaml-конфиге - методы.
В yaml-конфиге должен быть описан каждый платежный метод по-отдельности, а также явно указан маппинг
метода-канала.
Делай ровно так, как уже есть в примерах от fonbet. Эти конфиги мы потом будем поддерживать и править вручную,
поэтому от тебя требуется строго придерживаться сложившегося формата, ничего не выдумывай, не оптимизируй и
не фантазируй.
По каждому методу требуется указать набор необходимых полей (даже если придется дублировать, потому что 
набор полей может быть одинаковым на целую страну), их типы данных и регулярки где потребуется.

В коде адаптера запили функцию resolveChannel которая на вход принимает название платежного метода на нашей стороне,
а на выходе отдает объект Channel соответствующего канала.

### Проброс полей везде и всюду

Все поля из сгенерированных DTO продублируй также в общий объект TransactionContext, который уже общий между разными
адаптерами. Проверь каких полей в нем недостает и добавь новые из наших DTO, все в одну кучу.

Также в OpenAPI схеме добавь эти же поля (как опциональные) в соответствующие ручки запросов
DepositStart и WithdrawalStart.
В реализации HTTP-сервера позаботься о том, чтобы эти поля корректно заполнялись и пробрасывались дальше,
заполнялись в транзакцию
и благополучно доходили до TransactionContext и до самого адаптера SmilePayz.

В методах Deposit и Payout устрой проверку того, чтобы нужные поля были заполнены для конкретного канала.

### Сгенерируй API-клиент для отправки запросов к SmilePayz для депозита и для выплаты.

Следуй примерам кода от SmilePayz а также нашим примерам.

### Реализуй методы адаптера Deposit и Payout

Обрати внимание, что у нас запрос от пользователя может прийти в одной валюте, а платежная система
для конкретной страны может требовать другую валюту.
Перед вызовом Api-клиента сделай конвертацию валюты из валюты запроса в требуемую платежным методом валюту.

### Реализуй метод адаптера PaymentMethods

Он должен вернуть список доступных методов для конкретной страны, указанной в UserContext.
Если никакой страны не указано, то вернуть пустой список.

### Реализовать обработчик колбеков

Там в колбеках приходит идентификатор платежа, тебе надо в рамках обработки его найти у нас и применить статус,
если он терминальный. Надо замаппить их статус с нашим и вызвать соответствующую анонимную функцию.
