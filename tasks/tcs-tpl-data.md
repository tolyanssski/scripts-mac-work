# Реализовать формирование данных для шаблонов и отправку уведомлений с шаблонами в проекте br-tcs

## Добавить генерацию notification_id в br-tcs

Там где происходит выполнение таски и формирование уведомления для отправки в кафку,
добавь в dto новое поле notification_id (точный формат названия в json подсмотри в проекте br-users - dto NotifierPrivateMessage).
Оно должно быть nillable uuid.
Заполняй его сразу же свежесгенерированным uuid, а также этот uuid сохраняй в Task.NotificationID и таску сохраняй в БД после отправки сообщения.

## Сборка данных для письма в зависимости от notification_template_kind из trigger_rule

Сейчас отправка скипается если notification_template_kind не заполнено.
А если заполнено, то дальнейшая логика не реализована, настало время ее реализовать.

Сделай отдельный пакет, в котором будет набор сервисов, реализующих общий интерфейс, 
которые будут заниматься поиском и сериализацией данных для конкретного notification_template_kind.
Также в этом пакете отдельный сервис-фасад, в нем публичный метод принимает на вход строку
с notification_template_kind, под капотом в зависимости от конкретного значения строки
дергает нужный сервис-сборщик данных из своего пула сервисов, и возвращать то что
вернул сервис-сборщик: сформированную json-строку.

Этот сервис-фасад пробрось зависимостью в tasknotifications/worker. А в воркере в методе handleTask дергай этот сервис-фасад и возвращаемое им json-значение положи в поле Data у PrivateMessage.

## Как реализовывать сервисы-сборщики для конкретных notification_template_kind

В файле `/Users/h1/projects/notice_templates.sql` лежит SQL dump текущего состояния таблицы notice_templates.
Тебе нужно в этом дампе найти все записи, у которых указано значение в поле kind.
Для каждого найденного значения kind запили в br-tcs собственный сервис-сборщик данных
(kind в дампе == notification_template_kind в br-tcs).

### В каждом сервисе-сборщике

Проанализируй поле содержимое поля sender_email_mjml у записи notice_template,
соответствующей текущему kind.
Если поле не заполнено, то сервис-сборщик для этого kind не делай, его 
поддержку реализовывать не надо.
Если хоть поле заполнено, то вытащи из него все плейсхолдеры, которые имеют вид стандартных
плейсхолдеров go templates, например как этот: `{{.SomeVariable}}`.
Тебе нужно собрать все используемые названия плейсхолдеров в sender_email_mjml, без "точек" в начале. Сформируй на основе этого списка плейсхолдеров
приватное dto, которое потом перед выходом из публичного метода сериализуешь в json и отдашь полученный json строкой. Важно: в сериализованном json названия полей должны 1-в-1 быть
как в плейсхолдерах шаблона: то есть SomeVariable должно в json-виде остаться именно SomeVariable (а не someVariable или some_variable) - никаких преобразований case в названиях полей не делай.

### Где и как искать сами данные для заполнения

Тебе надо сначала по названиям самих плейсхолдеров оценить, что именно там требуется.
Как правило нужные данные лежат в других проектах, к которым надо ходить по grpc api,
в них эти данные либо в полях сущностей, либо в конфиге.

Пройдись по всем br-проектам в папке `/Users/h1/projects`. Посмотри где какие данные есть,
и если нужное значение ты нашел в конфиге другого проекта, то просто скопируй это значение
из того конфига и положи в конфиг br-tcs.

А если эти данные лежат в БД-сущностях проектов, то для доступа к ним потребуется сделать соответствующий запрос по grpc (как правило с поиском по user_id который тебе известен, либо также дополнительно по account_id, если он есть в конкретном событии, которое привязано к сработавшему триггеру), проанализируй GRPC API и то как проекты ходят друг к другу
по GRPC за данными, сделай так же.
Если требуется расширить GRPC API недостающими методами/полями, то вноси правки в соответствующие файлы в проекте br-contracts, который импортируется как библиотека
во все остальные проекты.

Если по названию плейсхолдера видно что это какое-то маркетинговое/информационное сообщение,
придумай для него текст на английском и запихни в конфиг и дергай оттуда.

Также: какие-то значения переменных следует брать из кастомного динамического json в ClientEvent.Metadata, из события по которому сработал триггер. Поскольку мы заранее не
знаем по какому именно типу события у нас сработал триггер (это настраивается в БД и может меняться), а также сами метаданные (их структура полей в json) зависит от конкретного события,
тебе нужно также проанализировать во всех проектах участки кода где эти события формируются 
и отравляются в кафка-топик tcs.events. Если там есть нужные поля с данными,
то тебе надо сделать проверки на eventType события сработавшего триггера, и если это тот
eventType который ты знаешь (видел его отправку в других проектах), то выбирай нужное поле.

Если каких-то данных ты в итоге совсем не нашел, но по смыслу все указывает на то,
что данные должны быть в сущностях какого-то конкретного проекта, то
пока никак их не заполняй, а сделай для их поиска отдельный метод-заглушку, в котором
потом реализуем grpc-запрос.
